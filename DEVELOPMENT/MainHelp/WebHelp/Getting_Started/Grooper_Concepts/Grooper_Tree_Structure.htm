Grooper 2.70 Documentation

Grooper Tree Structure

A Grooper repository is represented as a hierarchical tree structure.  At the root of this tree structure is the root node, and immediately below the root is a hierarchy of predefined folders which organize Grooper objects by functionality, as shown in the image to the right.  For example, the 'Batch Processing' folder, as its name implies, contains objects related to the processing of batches, such as Batches, Control Sheets, Batch Processes, and etc.  The predefined folders at the root of the Grooper tree structure are read-only, and cannot be modified, moved, or renamed.  

The tree structure is made of Grooper Objects, sometimes called "nodes", which represent everything from a page within a batch to an image cleanup profile.  Administrators configure Grooper by using Grooper Administration to browse and edit objects in the tree structure.  End users working in batches are also editing objects in the Grooper tree structure.  

Each folder in Grooper is designed to store specific object types.  For example, the 'Data Types' folder will only allow Data Type objects to be created or stored inside of it, with the exception that subfolders can be created to organize the Data Types.

Grooper Objects

All Grooper objects are derived from a common base object definition which defines the properties and behaviors common to all Grooper objects.  Some of these properties are as follows:

Name

All Grooper objects have a name, and the name of an object must be unique within the scope of the parent object.  Two objects with the same name may not exist in the same parent folder.

Attributes

Each Grooper object has a set of attribute flags, defined as follows:

Attribute Name Attribute Description
IsReadOnly Indicates that the item cannot be edited by users.
ContentsReadOnly Indicates that users cannot remove or add child items.
FullReadOnly This flag is a combination of IsReadOnly and ContentsReadOnly.
ContentsSorted If set, the contents of the item will be sorted alphabetically whenever items are added or renamed.
System The object is a system object.  System objects are read-only and cannot be modified, deleted, or renamed.
Has ACL Indicates whether an Access Control List has been created for the item.

Access Control List

Every Grooper object can have an Access Control List (ACL) which defines a list of users and groups, along with the security rights each of those entities has to the object.  Objects inherit the ACL of their parent object, unless the parent ACL is overridden by defining an ACL for the child object.  ACLs are created and managed on the Security Tab of the Node Properties Tab.

Properties

Each Grooper object type has custom properties which apply only to that type.  For example, a CMIS Connection object stores a URL, a user name, and a password for the target CMIS server.  These properties are specific to each object type, and are documented in the help topics for each object type.

Files

Every Grooper object has the capability to store any number of files.  For example, Batch Page objects store a "primary image" which is used to display the page, a thumbnail image, and may also store various other versions of the image created during Batch Processing.  OCR results extracted from the page are also stored as files on the Batch Page object.  Scripts are also stored as files on the object they are associated with.  A list of all files associated with a node can be viewed using the Files Tab of the Node Properties Tab.

Values

Each object also has a collection of name-value pairs which are used by activities and scripts to store information associated with that node.  For example, the Read Barcodes IP Command stores a serialized list of BarCodeInstance objects in a value on BatchPage objects named "Barcodes".  Third parties are free to store information in the values collection, but should avoid storing large amounts of data and should avoid using value names which begin with "Grooper".
