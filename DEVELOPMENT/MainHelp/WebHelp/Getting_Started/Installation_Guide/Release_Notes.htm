Version 2.40

The following sections outline the changes made in version 2.40

 

Script Editing Classification

Editing scripts with Visual Studio now requires Visual Studio 2013 (replacing Visual Studio 2012)

Electronic Document Processing

Ability to point Grooper to a repository of documents, render them into format for OCR

Case Uses:

  • Review large document repositories for PPI (Private Personal Information), Credit Card, SSNs

  • Search for specific types of documents (invoices, sales orders, etc)

  • Classify files and migrate them to another storage location with proper categorization

Batch Processing Redesign

Processing modules have been replaced with Activities

    • Previous modules were complex and hard to create. Most tasks didn't require an entire application to perform

    • Activities are lightweight - easy to create and maintain and custom activities can be created in minutes.

    • Custom unattended activities replace Processing Scripts

Span Batch activity - ability to move/copy objects from a current batch into a new batch based upon definable criteria

Ad-Hoc Task Processing

Multithreaded execution of any unattended activity that can be used only with Test batches.

    • Replaces 'OCR Test Batch dialog

    • Found in Grooper Admin under 'Tools / Process Tasks'

Improved Batch Process Designer

    • Real-time validation on all steps in the process

    • Configure steps from a single interface - no more popup windows

    • Improved documentation on all activity types and their properties.

Intensive Validation on Publish

    • The designed UI only validates the Batch Process and its steps

    • During publishing, all objects referenced in the activity's settings are also validated

New Classification Methods

Content Models can now be configured for Lexical or Visual Classification

Lexical Classification

    • Existing training-based classification

    • Now supports rules-based classification also, using rules defined on individual document types

Visual classification

    • Classifies documents by how they look. Uses same ESPâ„¢ training/learning mechanisms

    • Does not require full text OCR

    • Extremely fast - can run in real time during scanning

Rules-Based Classification

    • Available when Lexical classification is in use

    • Rules are defined on individual document types

    • Rules are specified as Grooper Data Types:

      • Positive Data Type - if the positive data type produces any results, the document will be classified as this type with no further processing

      • Negative Data Type - If the negative data type produces any results, this document type will be excluded as a candidate

    • Can be used stand-alone, or to handle exceptions with standard lexical classification

Extract Features IP Command

    • Divides an image into NxN matrix of cells, and generates 1 feature for each cell.

    • Features are used for training and classification, similar to keyword features in lexical classification.

    • Features can be based on a variety of measures such as cell intensity, gradient angle, COG angle, and COG magnitude.

Intensity Features

    • Each document layout has its own unique signature

    • ROI can be restricted to a specific area of the page

    • Grooper does not need much detail unless the documents are similar

COG Angle Features

    • Measures the angle between the center of each dell and the "center of gravity" for the black pixels.

    • Essentially, for each cell, Groper is learning "which way is the text?"

    • Can also generate features for COG Magnitude.

Scan-Time Classification & Separation

    • Using visual classification, Grooper can separate & classify documents in real time during scanning.

    • Works on structures documents only

    • Eliminates the need for separator sheets in cases where each file starts with a standard cover page.

Separation Profile Redesign

Dynamic Expressions

    • Simple alternative to Data Model scripting.

    • Are defined on Data Field objects.

    • Expression Types:

      • Default Value Expression â€“ Provides the initial value for the field.

        • Establishes the initial value for a field.

        • Must evaluate to the initial value desired for the field.

        • The value produced must be convertible to the fieldâ€™s base type.

        • Provides significantly more flexibility than a static list of variables.

      • Calculate Expression â€“ Evaluates to the expected value for the field.

        • Must evaluate to the expected value for the field.

        • Multiple Calculate Modes:

          • Validate: The expression is used to validate the field value.

          • SetIfEmpty: If the field is empty, it will be populated with the calculated value. Otherwise, the existing value will be validated.

          • AlwaysSet: The field value will be always be set to the calculated value.

        • Includes summing capabilities for Sections and Tables.

      • Validate Expression â€“ Evaluates to a boolean which indicates whether the field value is valid.

        • If the expression evaluates to false, the field is set to error using the Validate Message.

        • If the expression evaluates to true, the field is set to OK.

        • Runs after script validation.

    • Are dynamically-compiled .Net code, and thus have a broad syntax and range of capabilities.

Rubberband OCR Enhancements

Batch Transfer Over CMIS

Version 2.1

The following sections outline the changes made in version 2.1.

New Functionality

  • Control Sheets.  Added Control Sheet object type, allowing special control sheets to be created in Grooper Administration and used during the scan process to control document separation, classification, image cleanup, and ScanOnceâ„¢ settings while the scanner is moving.  A new batch Control Sheet generation tool was also added to the property panel for Content Model objects, allowing Grooper control sheets to be generated for each Document Type in the model.

  • Project Accounting.  Added Project object type to allow project-based stat tracking and cost accounting across multiple batch processes.  A new Project Billing Report is included which generates project-based cost accounting reports.  Batch Processes now have a new property to indicate which project they are associated with.

  • Cascading Zonal Profiles.  Zonal data extraction has been expanded so that zonal profiles can be created at any level of a Content Model.  This allows zonal recognition to be cascaded from broadly-defined zones at the model or category level, to more specific, tightly-defined zones at the document type and form type level.  This also provides the architectural groundwork for storing and using learned or user-created zones at any level of the Content Model.

  • User-Directed Zone Learning.  After a successful Rubberband OCR operation in the Index Panel, users may direct Grooper to learn the zone.  If the region overlaps with an existing zone, the two zones will be combined into one zone covering the union of the two regions. Otherwise, a new zone will be created for the field. This lays the groundwork for a feature in the next version where Grooper will automatically learn zones from successful ESP extractions, and use these to backup the ESP data extraction process.

  • Document-Level Field Registration.  Added support for document-level Field Registration Zones.  Previously, field registration was strictly at the page level - meaning that Grooper would search for the registration text within a specific zone on a specific page of the document.  With a document-level registration zone, no page number or zone is specified.  Grooper simply finds the registration text or pattern anywhere within the document, and aligns the zone relative to that location. If multiple matches are found, an extraction will be attempted on each instance, and all additional results will be returned as alternate field candidates.

  • Content Type Separation.  Separation Profiles can now separate based on the presence of a specific document type.  For example, this allows Grooper to separate batches of medical records based on the presence of a face sheet, eliminating the need to insert patch code separators or control sheets.  To use this feature, simply train Grooper with an example of the document to be used as a separator, and create a separation profile based on that document type.  Run an instance of Grooper Classification at the page level, followed by an instance of Grooper Separation running at the level above the pages, and configured with the separation profile created above.

  • Shape Detection.  A new IP Command named Shape Detection allows shapes such as stamps and other recognizable sub-images to the detected on pages in a batch.  The command can detect the location and orientation of the shape, and the results can be used by Separation Profiles to separate documents based on the presence of a stamp, or by a Field Zone to align the zone over the detected location of the stamp.

Miscellaneous Enhancements

  • Added support for regular expression syntax in the Registration Text property of a Registration_Zone.  Registration can now be based off of any pattern as well as any text string.
  • Added export and print functionality to the Page Viewer control.
  • Added functionality to Separation Profile objects to allow generation and printing of the separator sheets defined in the profile, and updated the Remove Foldering command to re-insert separator sheets.
  • Added Label Width and Label Alignment properties to all Data Model elements.
  • Added a Find command to the main menu of Grooper Administration, allowing any object in the Grooper tree structure to be located by name, type, or ID.
  • Added Back and Forward navigation in Grooper Administration.
  • Miscellaneous improvements to data extraction architecture.
  • Updated the Page Viewer control to support more export formats for images.

Bug Fixes

  • Fixed rubberband OCR issue where an exception would be thrown if the zone was exceptionally small.
  • Fixed an issue with drag and drop in Grooper Administration where an exception would be thrown during the DragOver event in certain cases.
  • Fixed an issue where the 'Enable Undo' checkbox on the settings page of the Image Processing Module was not properly enabled.
  • Fixed an issue where the Event Viewer Tab in Grooper Administration was ignoring the Procedure Name filter.
